# 组件示例编写提示词

## 1. 文件结构规范

### 父组件目录结构
在组件目录下创建 `doc/` 目录，包含以下文件：
- `api.md` - 组件API文档（表格格式：属性名|说明|类型|默认值）
- `summary.md` - 组件简要描述
- `example.json` - 示例配置文件（核心）
- `base.js` - 基础示例代码
- `[其他示例].js` - 扩展示例代码（如`transcript.js`、`scene.js`等）
- `style.scss` - 示例样式（可选）

### 子组件示例规则
**重要**：如果一个组件导出了多个子组件，所有子组件的示例都应该放在父组件的 `doc/` 目录中，而不是在子组件目录下创建单独的 `doc/` 目录。

示例：
```
src/components/Client/
├── index.js
├── Home/
│   └── index.js
├── Dashboard/
│   └── index.js
├── Learn/
│   └── index.js
├── Profile/
│   └── index.js
└── doc/
    ├── base.js          # Client主组件示例
    ├── home.js          # Home子组件示例
    ├── dashboard.js     # Dashboard子组件示例
    ├── learn.js         # Learn子组件示例
    ├── profile.js       # Profile子组件示例
    ├── example.json     # 所有示例配置
    ├── api.md           # 所有API文档
    ├── summary.md       # 组件描述
    └── style.scss
```

在 `example.json` 中为每个子组件添加独立的示例条目，子组件示例通过解构父组件导出进行引用：
```javascript
const { Home, Dashboard, Learn, Profile } = _Client;
```

## 2. example.json 配置结构
```json
{
  "isFull": false,  // 是否全屏展示
  "list": [
    {
      "title": "示例标题",
      "description": "示例说明",
      "code": "./base.js",  // 相对路径引用示例代码
      "scope": [  // 示例依赖包声明
        {
          "name": "_Report",  // 导入变量名（带下划线）
          "packageName": "@components/Report"  // 包路径
        },
        {
          "name": "_mockPreset",
          "packageName": "@root/mockPreset"
        },
        {
          "name": "remoteLoader",
          "packageName": "@kne/remote-loader"
        }
      ]
    }
  ]
}
```

## 3. 示例代码规范
- **导入方式**：使用 `scope` 中声明的变量名（带下划线前缀）
- **组件引用**：`const { default: ComponentName } = _ComponentName;`
- **工具包引用**：`const { createWithRemoteLoader } = remoteLoader;`

### 标准示例模板（需要远程加载和数据模拟）
```javascript
const { default: YourComponent } = _YourComponent;
const { default: preset, mockData } = _mockPreset;
const { createWithRemoteLoader } = remoteLoader;

const BaseExample = createWithRemoteLoader({
  modules: ['components-core:Global@PureGlobal']
})(({ remoteModules }) => {
  const [PureGlobal] = remoteModules;
  return (
    <PureGlobal preset={preset}>
      <YourComponent data={mockData.data} />
    </PureGlobal>
  );
});

render(<BaseExample />);
```

### 简化示例模板（无需远程加载）
```javascript
const { default: YourComponent } = _YourComponent;
const { Flex, Switch } = antd; // ⚠️ 使用了 antd 组件必须解构导入
const { useState } = React;

const BaseExample = () => {
  const [state, setState] = useState(false);
  return (
    <Flex vertical gap={12}>
      <YourComponent prop={value} />
    </Flex>
  );
};

render(<BaseExample />);
```

### 示例代码导入检查清单
编写示例代码时，请务必检查：
- [ ] 是否使用了 antd 组件（如 Space、Card、Button、Typography 等）？
  - 如果是，是否在代码中添加了 `const { 组件名 } = antd;` 解构导入？
  - 如果是，是否在 example.json 的 scope 中添加了 `{"name": "antd", "packageName": "antd"}`？
- [ ] 是否使用了其他项目组件（如 Icon、Modal 等）？
  - 如果是，是否在代码中添加了 `const { default: 组件名 } = _ComponentName;` 导入？
  - 如果是，是否在 example.json 的 scope 中添加了对应的依赖声明？
- [ ] 是否使用了 React Hook？
  - React Hook（如 useState、useEffect）无需导入，直接使用即可

## 4. scope 依赖声明规则
| 场景 | name | packageName |
|------|------|-------------|
| 项目组件 | `_ComponentName` | `@components/ComponentName` |
| Mock预设 | `_mockPreset` | `@root/mockPreset` |
| 远程加载器 | `remoteLoader` | `@kne/remote-loader` |
| Antd组件 | `antd` | `antd` |
| React原生 | `React` | -（无需声明，示例环境中默认可用） |
| 静态资源 | `bg` | `@components/Component/doc/img.png` |

**重要说明**：
- React 原生 API（如 `useState`、`useEffect`、`useRef` 等）在示例环境中默认可用，无需在 scope 中声明
- 在示例代码中直接使用 `React.useState` 或 `useState` 即可，不需要额外的导入
- React 相关的包（如 `react-icons`）如果需要在示例中使用，建议使用简单的 emoji 或文本替代

**⚠️ 特别注意：Antd 组件引入规范**
- 如果示例代码中使用了任何 antd 组件（如 `Space`, `Card`, `Button`, `Typography` 等），**必须**在 example.json 的 scope 中声明 `antd` 依赖
- **必须**在示例代码文件顶部添加 `const { 组件名 } = antd;` 解构导入语句
- 常见易错点：示例中使用了 antd 组件但忘记在 scope 中声明，或忘记在代码中解构导入，导致 "antd is not defined" 错误
- 只要示例中有一个组件使用了 antd，就必须在 scope 中声明 antd，即使其他示例没有使用

## 5. 数据准备
- 在 `mockPreset/index.js` 中导出mock数据
- 示例中使用 `import` 引入：`import reportDetail from './report/detail.json'`
- 通过 `preset` 传递给 PureGlobal 实现数据模拟

## 6. 组件开发要求
- 使用 `createWithRemoteLoader` 包装组件
- 在 `modules` 中声明远程依赖模块
- 通过 `remoteModules` 解构获取依赖组件

## 7. 示例内容设计原则

### 核心原则

**示例要根据API展示大部分功能，而且需要符合实际业务场景，示例数据也要尽量真实**

这是示例编写的核心要求，所有示例必须满足以下三点：

1. **API覆盖率**：示例必须覆盖组件API文档中的大部分功能，确保每个属性、方法都有对应的示例展示
2. **真实业务场景**：示例数据和使用场景必须贴近真实业务，避免使用"内容1"、"测试数据"等无意义的占位符
3. **数据真实性**：使用真实的人名、部门名称、业务状态等，让示例更具参考价值和可复制性

### 示例规划流程
1. **分析组件功能**：阅读组件源码，理解组件的核心功能和主要使用场景
2. **阅读 API 文档**：仔细阅读组件的 API 文档（`doc/api.md`），了解所有可用的属性、方法和配置项
3. **确定示例类型**：根据组件特性和 API 文档规划不同场景的示例，确保覆盖主要使用方式
4. **设计真实场景**：每个示例都应该模拟真实的业务场景，使用贴近用户的数据
5. **避免通用占位**：避免使用"内容1"、"示例数据"等通用占位符

**重要：参考 API 文档设计示例**

示例设计必须以 API 文档为依据，确保示例能够充分展示组件的所有可用 API。具体要求：

- **属性覆盖**：每个常用的属性都应该有对应的示例展示其效果
- **参数组合**：展示不同属性组合使用的效果，帮助用户理解属性之间的交互
- **默认值对比**：对比使用默认值和自定义值的差异
- **边界情况**：展示空值、无效值等边界情况的处理方式
- **可选功能**：对于可选的功能（如链接、图标等），分别展示开启和关闭的效果

**示例与 API 的对应关系检查表**：

在编写示例时，应该建立以下对应关系：
- 每个必填属性 → 至少有一个示例展示其必需性
- 每个常用可选属性 → 至少有一个示例展示其效果
- 每个特殊属性（如 className、style 等） → 有独立示例展示定制能力
- 每个复杂属性（如对象、数组、函数等） → 有详细示例展示其用法
- 每个枚举配置 → 有示例展示配置方式

### 示例命名规范
- 使用语义化的文件名，清晰表达示例的核心内容
- 示例标题简洁明了，突出该示例的亮点
- 示例描述说明该示例的使用场景和展示的功能

### 示例类型规划
根据组件特性，可能需要以下类型的示例：

| 示例类型 | 说明 | 示例场景 |
|---------|------|---------|
| 基础用法 | 展示组件最基本的使用方式 | 简单的数据展示、基础配置 |
| 功能特性 | 展示组件的核心功能和特性 | 表单编辑、数据加载、交互操作 |
| 参数配置 | 展示不同参数的配置效果 | 尺寸变化、样式定制、行为控制 |
| 高级用法 | 展示组件的高级功能或组合使用 | 自定义操作、复杂交互、数据联动 |
| 实际场景 | 模拟真实业务场景 | 用户详情、项目管理、订单处理等 |

### 数据真实性要求
- **使用真实姓名**：如"张三"、"李四"等，避免"name1"、"用户A"
- **合理业务数据**：部门用"技术部"、"产品部"，不用"部门1"、"部门2"
- **有意义的内容**：描述文字应该表达真实的业务含义
- **头像使用 dicebear**：如需头像，使用 `https://api.dicebear.com/7.x/avataaars/svg?seed=xxx`

**错误示例 ❌**：
```javascript
const data = [
  { label: "内容1", content: "这是内容1的描述" },
  { label: "内容2", content: "这是内容2的描述" },
];
```

**正确示例 ✅**：
```javascript
const data = [
  { label: "前端开发组", content: "负责所有前端页面开发和维护工作" },
  { label: "后端开发组", content: "负责API开发和服务器架构设计" },
];
```

### API 模拟规范
使用 DrawerButton 或 FetchButton 的示例，数据加载应该模拟真实的异步操作：
```javascript
api={{
  loader: () => {
    return new Promise((resolve) => {
      setTimeout(() => {
        resolve(realData);  // 返回真实的业务数据
      }, 500-1000);  // 模拟合理的网络延迟
    });
  },
}}
```

## 8. 示例完整性检查

### 示例覆盖度要求
在编写完示例后，需要检查以下内容是否被充分覆盖：

#### 必须覆盖的内容
1. **主要组件**：所有导出的主要组件都应该有对应的示例
2. **核心功能**：组件的核心功能点需要通过示例展示
3. **所有 API 属性**：根据 API 文档，每个属性都应该在至少一个示例中展示其效果
4. **常用属性**：常用的属性和配置项需要通过示例展示效果
5. **典型场景**：至少包含 1-2 个真实的业务场景示例

#### 补充示例的原则
如果现有示例不足以完整演示 API，应该补充新的示例文件。判断标准：
- 有子组件没有独立的示例
- 有 API 文档中列出的属性没有在示例中展示
- 有重要的 API 参数没有在示例中展示其效果
- 有特殊使用场景（如自定义字段、组合使用）没有示例
- 示例代码过于复杂，一个文件展示了太多功能，需要拆分
- **API 覆盖度不足**：对照 API 文档检查，有未展示的属性或功能

#### 补充示例的步骤
1. **分析 API 文档**：仔细阅读 `doc/api.md`，列出所有属性、方法和配置项
2. **对照现有示例**：检查每个 API 项是否已有示例展示，标记未覆盖的内容
3. **设计新示例**：为每个未覆盖的 API 功能点设计独立的示例文件
4. **示例文件命名**：应该清晰表达其展示的功能（如 `link.js`、`custom-style.js`）
5. **更新配置**：更新 `example.json`，添加新的示例条目
6. **验证完整性**：确保示例标题和描述准确表达示例内容，且能展示对应的 API

#### 示例文件数量建议
- **简单组件**：1-3 个示例文件即可
- **中等复杂组件**：3-6 个示例文件
- **复杂组件**（如 Filter）：6-10 个示例文件，覆盖各种子组件和高级用法

#### 示例完整性检查清单
在完成示例编写后，使用以下清单进行检查：

- [ ] 所有导出的主要组件都有对应的示例
- [ ] **所有 API 文档中列出的属性都有示例展示**
- [ ] **每个常用属性都有独立示例展示其效果**
- [ ] **每个可选配置（如链接、样式等）都有对比示例**
- [ ] 至少有一个基础用法示例
- [ ] 至少有一个真实业务场景的示例
- [ ] 有组合使用或高级用法的示例（如果适用）
- [ ] 示例文件命名清晰，能准确表达功能
- [ ] 示例标题和描述准确表达示例内容
- [ ] `example.json` 中的配置正确且完整
- [ ] 所有示例代码都能正常运行
- [ ] **对照 API 文档，确保没有遗漏的重要功能**

## 9. FormInfo 组件示例特殊规则

### 必须包裹 Form 避免上下文缺失
**重要**：如果编写的示例使用了 FormInfo 的字段组件（如 `Input`、`TextArea`、`SuperSelect` 等），而该组件在实际使用中没有在 `Form` 或 `FormModal` 中包裹，编写示例时必须自行添加外层 `Form`，否则会因为上下文缺失而报错。

### 标准 FormInfo 示例模板
```javascript
const { default: YourComponent } = _YourComponent;
const { createWithRemoteLoader } = remoteLoader;

const BaseExample = createWithRemoteLoader({
  modules: ['components-core:FormInfo', 'components-core:Global@PureGlobal']
})(({ remoteModules }) => {
  const [FormInfo, PureGlobal] = remoteModules;
  const { Form } = FormInfo;
  const { Input, TextArea } = FormInfo.fields;

  return (
    <Form onSubmit={(data) => console.log(data)}>
      <YourComponent />
    </Form>
  );
});

render(<BaseExample />);
```

### 错误示例 ❌
```javascript
// ❌ 错误：缺少外层 Form，会导致上下文缺失报错
const BaseExample = createWithRemoteLoader({
  modules: ['components-core:FormInfo']
})(({ remoteModules }) => {
  const [FormInfo] = remoteModules;
  const { Input } = FormInfo.fields;

  return (
    <YourComponent>
      <Input name="test" label="测试" />
    </YourComponent>
  );
});
```

### 正确示例 ✅
```javascript
// ✅ 正确：添加外层 Form 避免上下文缺失
const BaseExample = createWithRemoteLoader({
  modules: ['components-core:FormInfo']
})(({ remoteModules }) => {
  const [FormInfo] = remoteModules;
  const { Form } = FormInfo;
  const { Input } = FormInfo.fields;

  return (
    <Form onSubmit={(data) => console.log(data)}>
      <YourComponent>
        <Input name="test" label="测试" />
      </YourComponent>
    </Form>
  );
});
```

### 判断是否需要添加 Form
需要添加外层 `Form` 的情况：
- 示例代码中直接使用了 `FormInfo.fields` 中的任何组件
- 组件内部使用了 `useFormContext` Hook
- 组件依赖表单上下文进行字段校验或数据绑定

不需要添加外层 `Form` 的情况：
- 组件本身就是一个完整的表单（内部已包含 `Form`）
- 组件仅用于展示，不涉及表单字段
- 组件已经在 `FormModal` 中使用
